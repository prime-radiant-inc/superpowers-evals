**⊘ cluster — slow, not stuck.** All three ⊘ ran to their configured
`quorum_max_time` wall with rich non-empty capture (33 / 437 / 321 tool calls;
the capture-sensitive `skill-called` / `tool-called Agent` checks all passed),
clean logs, and the agent caught actively mid-task (writing a plan / dispatching
a fix-or-review subagent). Git history is monotonic — genuine forward progress,
no repetition. The slowness is structural: kimi's SDD loop multiplies its
per-call latency across ~5 subagent round-trips per task (implementer → spec
review → quality review → fix → re-review), running ~2–3× slower than the
scenario budgets assumed. This is the same family as antigravity's
heavyweight-workflow timeouts — but note the two SDD scenarios were **already at
90m**, so a modest budget bump will not rescue them (they need hours, or a
lighter task count).

**Skill-triggering signature — "under-invokes the gate."** Where codex *loads*
the gating skill then disobeys it, kimi **loads fewer skills and skips the gate
entirely**: it invoked no skill at all on `triggering-writing-plans`, and on
`worktree-caller-consent-gate` it loaded `executing-plans` but never followed the
hand-off to `using-git-worktrees`. It recognizes relevant skills in its own
reasoning but doesn't reliably *invoke* them or chase skill-to-skill hand-offs.

**Collaboration-layer signature — shared with antigravity.**
`worktree-consent-flow` is a near-identical fingerprint to antigravity's B3 fail
([`../2026-06-09.md`](../2026-06-09.md) lines 101–119): same `AskUserQuestion`
wording, AC2 satisfied, proceeded only after the scripted redirect. Kimi joins
antigravity (and old-claude on 05-29) in the cross-agent **consent-inference**
cohort — it can't infer that naming the worktree skill *is* consent. Combined
with `cost-tool-result-bloat` (no grep-first investigation discipline), kimi
shows the same "mechanics-right, collaboration-/discipline-layer-weak" family as
antigravity, plus its own under-invocation of gates.
