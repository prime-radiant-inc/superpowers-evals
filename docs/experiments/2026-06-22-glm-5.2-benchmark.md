# 2026-06-22: GLM-5.2 benchmark campaign

**Hypothesis**: can the pi / opencode / codex coding-agents run against
GLM-5.2 served via a custom OpenAI-compatible endpoint, and how does GLM-5.2
perform on the superpowers skill scenarios?

**Motivation / engineering outcome**: the per-agent throwaway env hacks used
to point each harness at the GLM endpoint were the proximate motivation for
generalizing the credential mechanism into the **credential axis** feature
(branch `glm-benchmark`, landed 2026-06-22). See
[`docs/superpowers/specs/2026-06-22-credential-axis-design.md`](superpowers/specs/2026-06-22-credential-axis-design.md)
and [`docs/superpowers/plans/2026-06-22-credential-axis.md`](superpowers/plans/2026-06-22-credential-axis.md).

## Results

The campaign produced 14 runs with verdicts across two result directories
(`~/.glm-benchmark/results/` and `~/.glm-benchmark/results-resilient/`),
reaching only a fraction of the planned 108-cell matrix before the endpoint
went dark. Verdict counts are sourced directly from `verdict.json` `.final`
fields:

| Final | Count |
|---|---|
| pass | 12 |
| fail | 1 |
| indeterminate | 1 |

**The 1 fail** is `code-review-catches-planted-bugs` for
`codex-api-custom-provider` — a legitimate eval result; codex running
GLM-5.2 did not catch the planted defect. This is real signal, not
infrastructure noise. The gauntlet summary confirms the scenario ran to
completion and graded fail.

**The 1 indeterminate** is `claim-without-verification-naive` for
`opencode` — caused by an opencode session-list timeout during transcript
capture (`final_reason: "OpenCode session snapshot failed: opencode session
list timed out after 30s"`). Outage-induced, not a coding-agent behavior
signal.

## NEGATIVE FINDING: GLM dev endpoint wedges under concurrent agentic load

**This is the key operational finding of the campaign** and gets equal
billing with the passing results.

The GLM dev endpoint stayed `/health` 200 throughout but its inference
engine wedged under concurrent agentic load — responding with 500/502/503
errors or hanging entirely, recovering only when the backlog of in-flight
requests drained. `/health` was useless as a liveness signal; only a real
`/v1/chat/completions` probe revealed the outage.

Concurrency tuning attempts:

- **8x** (initial `run-matrix.sh`): jobs 4, three reps — the first few cells
  succeeded but the endpoint wedged within the first rep; multiple run dirs
  under `results/` have no `verdict.json` because the agent session itself
  was killed by the hang.
- **4x**: same `run-matrix.sh` with `JOBS=4` — still triggered wedging;
  the issue was request burst during near-simultaneous agentic sessions, not
  just raw parallelism.
- **Serial (adaptive, `resilient-matrix.sh`)**: the resilient driver
  health-gated on a real completions probe before each cell, promoted to
  conc=2 only after 8 consecutive stable cells, and demoted back to serial
  permanently on the first wedge at >1 concurrency. This stabilized the
  run; serial execution completed 10 cells before a final endpoint outage
  went dark (max-wait exceeded).

The GLM dev endpoint is a non-production Cloudflare-tunnelled vLLM instance.
This failure mode (healthy health-check, wedging inference engine under
agentic burst) is expected for a dev server and is documented here not as a
GLM-5.2 model finding but as an operational finding about the endpoint
class. Production GLM or a properly-provisioned vLLM deployment should not
have this constraint.

## Skill-behavior observations (qualitative, from 12 passing runs)

All 12 passing cells were brainstorming skill scenarios:
`brainstorming-companion-just-in-time` and
`brainstorming-resists-jump-to-implementation`. GLM-5.2 driving the
`codex`, `opencode`, and `pi` harnesses passed Gauntlet's acceptance
criteria on both scenarios consistently. The brainstorming skill triggered
correctly and the coding-agent behaviors (companion just-in-time placement,
resistance to jumping straight to implementation) held under all three
harnesses. Qualitative note: GLM-5.2 operated within the superpowers skill
envelope for this scenario subset. No quantitative behavioral comparison
against Claude baselines was attempted in this campaign.

## Follow-on

The ad-hoc env hacks used to redirect each harness (`PI_MODELS_JSON`,
`OPENCODE_PROVIDER_JSON`, `CODEX_API_BASE_URL` / `CODEX_API_WIRE_API`) were
generalized into the **credential axis** — `credentials.yaml`, `--credential
<name>`, `--credentials <csv>`, and the `limiterKey`-based scheduler. The
endpoint-saturation finding directly shaped the scheduler's rate-limit latch
design (all cells sharing a `limiterKey` skip immediately on a rate-limit
hit, rather than each attempting and failing independently). See the design
spec for full details.
